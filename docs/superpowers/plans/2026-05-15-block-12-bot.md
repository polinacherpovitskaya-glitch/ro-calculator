# Block 12 — Telegram task bot — переключение + redesign для стабильности

> **REQUIRED:** Block 11 (tasks таблицы готовы) + **[BUG-INVENTORY.md](2026-05-15-BUG-INVENTORY.md)** классы P/Q/R + **[STABILITY-PROGRAM.md](2026-05-15-STABILITY-PROGRAM.md)**.
>
> Пользователь сказал: *«часто падает бот для записи часов»*. Это симптом класса P (state в файлах теряется при рестарте). Block 12 — это **не только перенос на Postgres**, это **архитектурная переделка бота** для устранения класса P/Q/R целиком.

**Goal:**
1. Существующий бот в `bot/` ходит в Supabase. Переключить на новый Postgres.
2. **Перенести state с локальных файлов (timebot.state.json, timebot.pending.json, timebot.inbox.jsonl) в БД.** Это устраняет класс P (restart loses state) полностью.
3. **Чистая модель telegram binding** через отдельную таблицу. Устраняет класс R (binding mess).
4. **Явная TZ-логика** через `employees.timezone` + `date-fns-tz`. Устраняет класс Q (date shift).

**Source reference:** каталог `bot/` (`taskbot.js`, `timebot.js`, `task-notification-worker.js`, `timebot-freeform-parser.js`, `timebot-state-utils.js`).

**Dependencies:** Block 11 (tasks в Postgres), Block 14 (time_entries в Postgres — если бот пишет туда).

-- 013_bot_state.sql
-- State бота переходит из локальных файлов в БД (фикс класса P).
-- Restart процесса теперь не теряет state.

-- Маппинг telegram_chat_id → employee. Один активный binding на сотрудника.
CREATE TABLE IF NOT EXISTS bot_telegram_bindings (
    telegram_chat_id   BIGINT PRIMARY KEY,
    telegram_username  TEXT,
    employee_id        INTEGER NOT NULL REFERENCES employees(id) ON DELETE CASCADE,
    bound_at           TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_active_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    is_active          BOOLEAN NOT NULL DEFAULT TRUE
);
-- Один активный binding на сотрудника (если new binding — старое деактивируется)
CREATE UNIQUE INDEX IF NOT EXISTS idx_bot_binding_active_employee
    ON bot_telegram_bindings(employee_id) WHERE is_active = TRUE;

-- Текущий state диалога с пользователем. Заменяет timebot.state.json.
CREATE TABLE IF NOT EXISTS bot_conversation_state (
    chat_id            BIGINT PRIMARY KEY REFERENCES bot_telegram_bindings(telegram_chat_id) ON DELETE CASCADE,
    flow               TEXT,                          -- 'time_entry', 'task_create', 'idle'
    step               TEXT,                          -- 'asking_hours', 'asking_project', ...
    draft              JSONB NOT NULL DEFAULT '{}',   -- частично собранные данные
    expires_at         TIMESTAMPTZ NOT NULL,           -- автоудаление через 24 часа неактивности
    updated_at         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_bot_state_expires ON bot_conversation_state(expires_at);

-- Очередь входящих сообщений (если бот обрабатывает предыдущее).
-- Заменяет timebot.inbox.jsonl и timebot.pending.json.
CREATE TABLE IF NOT EXISTS bot_message_inbox (
    id                 BIGSERIAL PRIMARY KEY,
    chat_id            BIGINT NOT NULL,
    telegram_message_id BIGINT NOT NULL,
    text               TEXT,
    payload            JSONB DEFAULT '{}',            -- весь telegram update'а целиком, на случай если text не хватает
    received_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    processed_at       TIMESTAMPTZ,
    processing_started_at TIMESTAMPTZ,
    error              TEXT,
    UNIQUE(chat_id, telegram_message_id)
);
CREATE INDEX IF NOT EXISTS idx_bot_inbox_unprocessed ON bot_message_inbox(chat_id, received_at) WHERE processed_at IS NULL;

-- Telegram токены для API доступа (бот → /api/* с Bearer)
CREATE TABLE IF NOT EXISTS bot_tokens (
    token         TEXT PRIMARY KEY,

    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    last_used_at  TIMESTAMPTZ
);

-- TZ-поле для сотрудника (фикс класса Q)
ALTER TABLE employees ADD COLUMN IF NOT EXISTS timezone TEXT DEFAULT 'Europe/Moscow';

INSERT INTO app_meta (id, version) VALUES (1, '013-bot-state')
ON CONFLICT (id) DO UPDATE SET version = EXCLUDED.version, applied_at = NOW();

## Task 2: Заменить Supabase-клиент на pg

В `ops/bot/` найти все места использования Supabase (`createClient` из `@supabase/supabase-js`, или прямые HTTP-запросы к Supabase REST). Заменить на наш Postgres через `pg`-driver или на наш `/api/` (через fetch).

Подход: использовать наш собственный API `/api/tasks`, `/api/auth`, и т.п. Тогда бот не имеет прямого доступа к БД — это секурнее.

Но: бот часто работает по `cron`-расписанию и может не иметь cookie-сессии. Решение:
- Добавить в БД таблицу `bot_tokens (token TEXT PRIMARY KEY, name TEXT, created_at TIMESTAMPTZ)`.
- В `ops/api/src/middleware/auth.js` поддержать заголовок `Authorization: Bearer <token>` параллельно с cookie.
- Бот шлёт `Authorization: Bearer <token>` где token прописан в `.env`.

Migration:

## Task 5: state.js — state в БД (фикс класса P)

**File:** `ops/bot/state.js`

```js
// Заменяет timebot.state.json, timebot.pending.json, timebot.inbox.jsonl.
// State хранится в БД, восстанавливается при рестарте.

import pg from 'pg';
const pool = new pg.Pool({ connectionString: process.env.DATABASE_URL });

const STATE_TTL_HOURS = 24;

export async function loadState(chatId) {
  const { rows } = await pool.query(
    `SELECT flow, step, draft FROM bot_conversation_state
     WHERE chat_id = $1 AND expires_at > NOW()`,
    [chatId]
  );
  if (rows[0]) return rows[0];
  return { flow: 'idle', step: null, draft: {} };
}

export async function saveState(chatId, state) {
  const expiresAt = new Date(Date.now() + STATE_TTL_HOURS * 60 * 60 * 1000);
  await pool.query(
    `INSERT INTO bot_conversation_state (chat_id, flow, step, draft, expires_at, updated_at)
     VALUES ($1, $2, $3, $4, $5, NOW())
     ON CONFLICT (chat_id) DO UPDATE SET
       flow = EXCLUDED.flow, step = EXCLUDED.step, draft = EXCLUDED.draft,
       expires_at = EXCLUDED.expires_at, updated_at = NOW()`,
    [chatId, state.flow || 'idle', state.step || null, state.draft || {}, expiresAt]
  );
}

export async function clearState(chatId) {
  await pool.query(`DELETE FROM bot_conversation_state WHERE chat_id = $1`, [chatId]);
}

export async function enqueueMessage(chatId, telegramMessageId, text, payload) {
  await pool.query(
    `INSERT INTO bot_message_inbox (chat_id, telegram_message_id, text, payload)
     VALUES ($1, $2, $3, $4)
     ON CONFLICT (chat_id, telegram_message_id) DO NOTHING`,
    [chatId, telegramMessageId, text, payload || {}]
  );
}

// Берёт следующее необработанное сообщение для chat_id, отмечает processing_started_at (lock).
// Если кто-то другой уже взял — возвращает null.
export async function takeNextMessage(chatId) {
  const { rows } = await pool.query(`
    UPDATE bot_message_inbox SET processing_started_at = NOW()
    WHERE id = (
      SELECT id FROM bot_message_inbox
      WHERE chat_id = $1 AND processed_at IS NULL AND processing_started_at IS NULL
      ORDER BY received_at LIMIT 1
      FOR UPDATE SKIP LOCKED
    )
    RETURNING *
  `, [chatId]);
  return rows[0] || null;
}

export async function markProcessed(messageId, error = null) {
  await pool.query(
    `UPDATE bot_message_inbox SET processed_at = NOW(), error = $2 WHERE id = $1`,
    [messageId, error]
  );
}

// Cron: удалить старые processed сообщения (> 7 дней)
export async function cleanupOldMessages() {
  await pool.query(
    `DELETE FROM bot_message_inbox WHERE processed_at IS NOT NULL AND processed_at < NOW() - INTERVAL '7 days'`
  );
}
```

Тесты в `ops/bot/test/state.test.js`:
- saveState → loadState возвращает сохранённое
- Два saveState подряд → loadState возвращает второе
- expires_at прошёл → loadState возвращает идемпотентный idle
- enqueueMessage + takeNextMessage → возвращает в FIFO порядке
- takeNextMessage с FOR UPDATE SKIP LOCKED: два параллельных вызова берут разные сообщения
