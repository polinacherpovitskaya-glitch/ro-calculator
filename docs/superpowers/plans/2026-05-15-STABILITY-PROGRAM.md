# Stability Program — что мы делаем системно, чтобы «летало и не падало»

> **Контекст:** Пользователь сформулировал цель: *«чтобы летало и никогда не падало»*. Это амбициозно. Этот документ — честный план как мы к этой цели приближаемся в каждом блоке миграции. Это не «обещание что багов не будет» — это набор защитных механизмов, которые **уничтожают целые классы багов** на уровне архитектуры.

> **Главная идея:** мы не правим баги по одному. Мы строим систему так, чтобы целые классы багов **технически не могли возникнуть**. Это сильно надёжнее ежедневной чинки.

---

## Семь системных защит

### 1. Single source of truth + один путь данных
**Что устраняет:** Cache hell (класс A), stale assets (O), bootstrap rot.
**Как:**
- Один путь: UI → API → Postgres. Никаких snapshot'ов, никаких «локальных кэшей с авто-синком», никаких bootstrap'ов от 30 минут назад.
- localStorage — **запрещён для бизнес-данных**. Только UI-state (тёмная тема, последний открытый таб) — не данные.
- После reload UI всегда читает свежее с сервера.
- Vite hash-naming для статики — нельзя получить старый JS после деплоя.

### 2. Idempotency-Key на каждой мутации
**Что устраняет:** двойное списание (B), двойная приёмка, draft duplicate saves (L), любые race retry'и.
**Как:**
- Middleware `withIdempotency()` (Block 3) обязателен на каждом POST/PATCH/DELETE.
- Клиент генерирует UUID на каждое действие (`crypto.randomUUID()`).
- Сервер хранит результат + статус в `idempotency_keys`. Повторный запрос с тем же ключом → тот же ответ, никаких побочных эффектов.
- TTL 24 часа — потом ключи чистятся.

### 3. Транзакции + `SELECT FOR UPDATE`
**Что устраняет:** race conditions (G), partial saves (Y), все «половина сохранилась».
**Как:**
- Все multi-step операции (consume-hardware, receive-shipment, mold-use, reservation-update) — в `withTransaction`.
- Lock takeовые items через `SELECT id FROM warehouse_items WHERE id = ANY($1) ORDER BY id FOR UPDATE`.
- При ошибке → ROLLBACK → 4xx/5xx с понятной error.code. UI знает что ничего не применилось.
- ORDER BY id в FOR UPDATE — избегаем дедлоков.

### 4. Foreign Keys + CASCADE
**Что устраняет:** фантомные резервы (C), orphan записи, order resurrection (S).
**Как:**
- `warehouse_reservations.order_id REFERENCES orders(id) ON DELETE CASCADE`
- `order_items.order_id REFERENCES orders(id) ON DELETE CASCADE`
- `bug_attachments.bug_id REFERENCES bug_reports(id) ON DELETE CASCADE`
- DELETE order = физическое удаление + CASCADE на зависимости. Никаких soft-delete с возможностью «возродиться».

### 5. TypeScript на всех границах
**Что устраняет:** numeric coercion (M), сюрпризы вроде «передал строку где ждали число».
**Как:**
- `ops/api/src/calc/` — целиком на TypeScript. Compile errors блокируют CI.
- `ops/web/src/` — целиком на TypeScript.
- API contract: request body валидируется через схемы (например `zod`). Если клиент шлёт строку вместо числа — 400 с понятным error message, не молчаливая обработка.
- Один источник типов: схемы в `ops/api/src/types.ts`, переиспользуются между API и Web.

### 6. State в БД, не в файлах/памяти
**Что устраняет:** restart loses state (P), bot crashes теряют данные, partial saves в боте.
**Как:**
- Бот (Block 12): conversation state в таблице `bot_conversation_state`, не в `timebot.state.json`.
- API stateless: каждый запрос приходит с auth cookie, не имеет процесс-локального state.
- Process restart → state в БД остаётся → продолжаем с того же места.

### 7. Snapshot semantics для финансовых данных
**Что устраняет:** цены drift (F), margin drift (U), factual recalc surprise (W).
**Как:**
- `orders.calculator_data` JSON — **снапшот** на момент последнего recalc.
- `order_factuals.factual_data` — снапшот на момент закрытия.
- Цены текущие — в `warehouse_items.last_price`, но они **не текут автоматически** в закрытые заказы.
- Юзер хочет обновить расчёт текущими ценами → явная кнопка «Пересчитать». Никакого автомата.

## Защиты на уровне разработки

### Тесты как защитная сетка

**Уровни тестов:**

1. **Unit-тесты модулей** (`node --test`)
   - Каждая чистая функция (calc формулы, payroll, factual aggregation) покрыта 5-10 тестами включая edge case'ы.
   - ≥ 300 unit-тестов на финальный проект.

2. **Integration-тесты API** (тоже `node --test` с временной Postgres)
   - Каждый эндпойнт — позитив, негатив, idempotency, auth, валидация.
   - ≥ 200 integration-тестов.

3. **Golden-master тесты** (Block 7)
   - 20+ реальных заказов, цифры байт-в-байт.
   - Защищают от регрессий в формулах. Если кто-то «улучшит» формулу и сломает один реальный заказ — CI красный.

4. **Инвариантные тесты** (Block 3)
   - I1-I7 SQL-запросы запускаются после каждого тестового сценария.
   - Если хоть один инвариант нарушен → fail.

5. **Playwright E2E**
   - 1-2 сценария на блок. Минимальный набор: login → ключевые операции → проверка.
   - Запускаются в CI после деплоя на staging.

6. **Cron инвариант-чекер на проде**
   - После Stage C: каждый час cron на VPS запускает I1-I7 на live-БД. Если нашёл нарушение — Telegram алёрт.

### Бюджет на тесты

| Категория | Время на блок |
|---|---|
| Unit | 20-30% времени блока |
| Integration | 20-30% времени блока |
| E2E + manual smoke | 10% времени блока |
| Golden-master (только Block 7+9) | 30% времени Block 7, 20% Block 9 |

Это значит **половина времени каждого блока — тесты**. Без этого «летает и не падает» не получится.

## Защиты на уровне инфраструктуры

### Health monitoring

**UptimeRobot (бесплатный)** пингует `/api/health` каждую минуту:
- 2 фейла подряд → email + Telegram алерт
- Endpoint возвращает `{ status, version, db: { ok, latency_ms }, uptime_seconds }`
- Если БД недоступна → `status: 'degraded'`, UptimeRobot видит и алёртит

**Cron инвариантов:** каждый час `node ops/scripts/check-warehouse-invariants.mjs` + при нарушении посылает Telegram.

**Postgres slow query log:** `log_min_duration_statement = 500`. Все запросы > 500 мс попадают в логи. Раз в неделю — обзор.

**Docker logs ротация:** `--log-opt max-size=10m --log-opt max-file=5` — не съест диск.

### Backups + restore drill

- Ежесуточный `pg_dump` → Selectel S3 (retention 30 дней)
- **Раз в месяц** — `scripts/restore.sh` на staging. Это **обязательная процедура**. Если бэкапы не проверены — они не работают.
- Бэкапы автоматизированы через systemd timer.

### Deploy / Rollback

- CI деплоит на staging автоматически на push в main
- Деплой создаёт тег `deploy-YYYYMMDD-HHMM`
- Откат: `scripts/rollback.sh <tag>` — переключает symlink на старую версию, `docker compose restart`. Реальное время отката < 2 мин.
- После деплоя — `curl /api/health` с retry. Если 5 попыток фейл → автооткат.

### Limits / quotas

- Postgres `statement_timeout = 30s` — никакая операция не висит дольше.
- API connection pool: 10 connections. Если все заняты — клиент получает 503, не висит.
- Node API memory limit: 512 MB (через docker), процесс перезапускается на OOM.
- Rate limit на login: 10 попыток / 15 мин на IP. Защита от brute-force.

## Защиты на уровне команды/процесса

### Только через PR

- main защищён: прямой push запрещён. Только через PR.
- CI должен быть зелёным перед merge.
- PR описание: какие баги класса A-Y из BUG-INVENTORY затронуты.

### «Не чиним по живому»

- Все изменения проходят через staging.
- В прод не пушим в пятницу вечером (если не cutover-окно).
- Не «чиним багу в проде SSH'ом» — только через нормальный deploy.

### Документация рядом с кодом

- `ops/README.md` — главный playbook (как deploy, как restore, как SSH)
- `ops/api/src/calc/README.md` — формулы калькулятора
- `WAREHOUSE-INTERACTION-MAP.md` — карта склада
- Этот документ — карта защит
- Каждый блок имеет свой план с acceptance criteria

## Реалистичные ожидания после миграции

**Что станет ЛУЧШЕ ГАРАНТИРОВАННО:**
- ✅ Никаких «висит при сохранении» (классы A, Y)
- ✅ Никаких «после reload старые цифры» (A)
- ✅ Никаких «дважды списалось» (B, L)
- ✅ Никаких «удалённый заказ возродился» (S)
- ✅ Никаких «бот забыл после рестарта» (P)
- ✅ Никаких «фантомных резервов» (C)
- ✅ Никаких «timezone сдвиг» (Q)
- ✅ Быстрая загрузка (sub-second) после Block 1
- ✅ Понятные ошибки с error.code вместо «что-то пошло не так»
- ✅ Деплой в любой день — нет страха «всё развалится»

**Что станет ЛУЧШЕ С РАЗБОРОМ:**
- ⚙️ Формулы калькулятора — golden-master гарантирует, что после миграции они работают КАК БЫЛО. Если в формулах есть баги — они там и останутся, надо их находить отдельно после Stage D.
- ⚙️ UX заказа — порт 1:1, никаких UX улучшений в рамках миграции. Улучшения после.
- ⚙️ Analytics — переносим с известными багами. Фиксим после Stage D.

**Что НЕ исчезнет автоматически:**
- ❌ Баги бизнес-логики (вроде «в pricing для оптовиков применяется не та маржа») — должны быть выявлены явно, иначе перенесутся.
- ❌ Баги в редких сценариях, не покрытых golden-master fixtures — могут проявиться позже. Поэтому Stage B = неделя ручного тестирования с менеджерами.
- ❌ Новые баги от изменений в новой системе. Поэтому тесты + Stage B + safety net в Stage D.
