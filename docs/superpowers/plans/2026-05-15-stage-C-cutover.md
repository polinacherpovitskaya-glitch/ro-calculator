# Stage C — Cutover Day Playbook

> **REQUIRED:** Stage B завершён со всеми зелёными чек-боксами.

**Goal:** За 2-4 часа в субботу/воскресенье вечером переключить production трафик со старой `calc.recycleobject.ru` (Vercel + Supabase) на новую `ops-staging.recycleobject.ru` (Selectel). После cutover — старая система остаётся в read-only режиме на ≥7 дней как safety net.

**Это самый важный день всего проекта.** Перед ним — репетиция (Task 1 ниже).

---

## Task 1: Репетиция cutover накануне

**Цель:** прогнать все шаги cutover'а на staging-копии без реального переключения DNS, чтобы убедиться что всё работает.

- [ ] За 24-48 часов до планируемого cutover, на staging-копии данных:
  - Прогнать финальный delta-sync скрипт (`refresh-staging-snapshot.mjs`)
  - Замерить время — должно быть ≤ 30 мин
  - Прогнать сверку (`compare-datasets.mjs`)
  - Прогнать golden master тесты
  - Прогнать Playwright smoke

- [ ] Зафиксировать в чате/доке (например `docs/cutover-rehearsal-results.md`): шаги, время, какие возникли проблемы.

- [ ] Если что-то пошло не так — починить, повторить репетицию. Только когда репетиция прошла гладко — назначать cutover.

---

## Task 2: T-24h — Подготовка

За 24 часа до окна cutover:

- [ ] Объявить ВСЕМ сотрудникам в чате (например в Telegram):
  > «Завтра в 22:00 МСК мы переключаемся на новую систему. С 21:30 до примерно 00:30 не делайте новых заказов / не списывайте фурнитуру / не меняйте задачи в `calc.recycleobject.ru`. Если очень срочно — пишите в чат. После 00:30 откроется доступ к новой системе на `ops.recycleobject.ru` (URL обновится у всех автоматически). Ваш логин — тот же email; если забыли пароль — пишите.»

- [ ] Подготовить env vars на VPS:
  - Все секреты должны быть на месте
  - `SUPABASE_SERVICE_KEY` доступен для финального sync
- [ ] Открыть Selectel-консоль (быстрый доступ если нужен)
- [ ] Открыть DNS-консоль reg.ru (для финального переключения)
- [ ] Открыть терминал на машине разработчика с активной SSH-сессией к VPS

- [ ] **Включить мониторинг побольше**: каждые 30 сек проверять `/api/health`. Если упадёт — алёрт.

## Task 3: T-30мин — Объявление

> «Через 30 минут начинаем переключение. Уже сейчас завершайте работу с заказами и складом. Не оставляйте незавершённых форм.»

- [ ] Проверить что в логах нет активных пользовательских сессий перед началом (опц).

## Task 4: T+0 — Перевод старой системы в read-only

В 22:00 МСК (или назначенное время):

- [ ] Включить kill-switch в старой calc.recycleobject.ru. Реализация — добавить флаг в `settings.readOnly = true` в Supabase, и в `js/supabase.js` (старого фронта) перед каждой мутацией проверять флаг. Если флаг включён — показать «Система в режиме обслуживания, попробуйте через 1 час».

- [ ] Альтернатива: на стороне Vercel-deployment — выставить environment variable `READ_ONLY=true`, и API возвращает 503 на все non-GET. Реализовать заранее (например в Block 16 как dry-run, не активируя).

- [ ] Проверить через curl что POST/PATCH на старую calc возвращают 503 / «read-only».

---

## Task 5: T+5 — Финальный delta-sync

Запустить полную перезаливку staging-БД из Supabase:

```bash
ssh root@ops-staging.recycleobject.ru
cd /srv/ops
docker exec ops-postgres psql -U ops -d ops -c "SELECT NOW()"  # проверить что БД доступна

# Запуск финального sync — берёт всё свежее из Supabase
SUPABASE_URL=https://jbpmorruwjrxcieqlbmd.supabase.co \
SUPABASE_SERVICE_KEY=<key> \
DATABASE_URL=postgres://ops:<password>@localhost:5432/ops \
  node ops/scripts/refresh-staging-snapshot.mjs 2>&1 | tee /tmp/cutover-refresh.log
```

- [ ] Ждать завершения. Логировать время.
- [ ] Если ошибка — **СТОП**. Решить вкл/откат.

## Task 7: T+45 — DNS переключение

В DNS reg.ru поменять:
- A `calc` → `176.114.65.153` (IP нового сервера) — было `76.76.21.21` (Vercel)
- CNAME `www` — если использовалось, добавить redirect на `calc.recycleobject.ru`

Подождать DNS propagation. TTL у нас 300 сек, должно прорасти за 5-15 мин.

- [ ] Проверить:
  ```bash
  dig +short calc.recycleobject.ru @ns1.reg.ru
  dig +short calc.recycleobject.ru @8.8.8.8
  ```
- [ ] Дополнительно: открыть https://calc.recycleobject.ru в инкогнито в браузере — должен открыться новый UI.

**Также:** добавить SAN/Alt-Name в Caddy конфиг чтобы он выпустил cert и для `calc.recycleobject.ru`. Перезапустить Caddy.

В `ops/infra/Caddyfile`:
```
{$DOMAIN}, calc.recycleobject.ru {
    ...
}
```

Установить `DOMAIN=ops-staging.recycleobject.ru` (старый) — Caddy выпустит TLS и для второго имени автоматически.

- [ ] Перезагрузить compose:
  ```bash
  docker compose --env-file .env restart caddy
  ```

- [ ] Проверить TLS на новом адресе:
  ```bash
  curl -I https://calc.recycleobject.ru
  ```

## Task X (Emergency): Откат

Если в любой момент cutover'а возникла критическая проблема:

```bash
# 1. Вернуть DNS обратно
# В reg.ru: A calc → 76.76.21.21 (Vercel)

# 2. Снять kill-switch со старой calc
# В Supabase: UPDATE settings SET value = 'false' WHERE key = 'readOnly';

# 3. Сообщить в чат: «Возвращаемся на старую систему, разбираемся.»
```

DNS обратно прорастёт за 5-15 мин. После этого всё снова работает как было. Через 5 минут все пользователи смогут продолжить работать на старой.

Найти баг → починить → новая попытка cutover на следующих выходных.

После 24 часов без блокеров — Stage C завершён. Переходим к Stage D.
