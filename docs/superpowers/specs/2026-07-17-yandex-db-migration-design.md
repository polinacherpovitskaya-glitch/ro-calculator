# Миграция базы на российскую инфраструктуру (Yandex Cloud)

**Дата:** 2026-07-17 · **Статус:** дизайн, ждёт подтверждения Полины
**Цель:** убрать последний зарубежный SPOF — Supabase (Postgres + Storage). После миграции ни VPN, ни «522», ни зависимость от зарубежных площадок для ежедневной работы.
**Жёсткое ограничение:** НЕ переписывать приложение (опыт ops/ не повторяем). Меняются только URL/ключи/конфиги.

**Потребители базы (всем поменять URL+anon key):** calc/calc2 (`js/supabase.js` — одно место), сайт recycle-object-site (env на Vercel + `next.config` домены картинок + media-proxy), CI-секреты (`SUPABASE_URL`, `SUPABASE_ANON_KEY` в двух репо), Яндекс-прокси (API Gateway upstream), floor-публикатор, timebot (env), **Figma-плагин** (конфиг у Элины — отдельный чекбокс!).

**URL файлов захардкожены в данных:** `photo_url/glb_url/template_url/grey_thumb_url` в mold_data и блобах указывают на `jbpmorruwjrxcieqlbmd.supabase.co` → нужен скрипт массовой замены домена в данных (шаг 5).

## Архитектура (рекомендация: вариант B)

**B — self-hosted Supabase (Docker) на Yandex Compute VM.** Официальный docker-compose (без Studio/Analytics/Realtime-профилей): Postgres + PostgREST + Storage API + Kong + GoTrue-минимум. Спереди Caddy с автоматическим TLS.
- ✅ **100% совместимость API**: тот же `/rest/v1/`, тот же `/storage/v1/object/public/...`, supabase-js работает как есть → в коде меняется ОДНА строка URL + anon key. Ровно то, что просила Полина.
- ✅ VM 2 vCPU / 4 ГБ / 50 ГБ SSD ≈ **1 800–2 500 ₽/мес** (Supabase $15/мес отпадает → почти в ноль).
- ⚠️ Обслуживаем сами → компенсация ДИСЦИПЛИНОЙ С ПЕРВОГО ДНЯ: nightly `pg_dump` → Yandex Object Storage (уже есть бакет и токены в CI), weekly снапшот диска, healthcheck-мониторинг с алертом в Telegram (переиспользуем существующий алерт-скрипт CI).

Отвергнутые:
- **A. Managed PostgreSQL + PostgREST + Object Storage напрямую** — надёжнее опсово, но Storage API придётся заменять своим upload-слоем (calc пишет фото через `supabase.storage`) = правки кода. Дороже (~3–4К ₽/мес).
- **C. Только прокси, база остаётся в Supabase** — не решает SPOF (522-инциденты были в самом Supabase). Нет.

**Домен:** `db.recycleobject.ru` → VM (DNS уже в Яндексе). Старый и новый URL взаимозаменяемы на время миграции.

## Ключи
Свой `JWT_SECRET` → генерим новые `anon`/`service_role` ключи. Замена идёт парой «URL + ключ» в тех же местах. Старый anon key перестаёт работать после сноса старого проекта (безопасность улучшается: сейчас anon пишет всё и всем).

## План (этапы, каждый обратим)

1. **Подготовка (без влияния на прод):** создать VM, домен, TLS, поднять compose, сгенерить ключи, smoke-тест пустого стека.
2. **Первичный перенос:** `pg_dump` со старого (нужен db-пароль из Supabase-дашборда) → restore; выгрузка 270 файлов Storage (публичные) → загрузка в новый Storage теми же путями.
3. **Стенд-проверка:** локально/на calc2-копии подменить URL → прогнать все смоуки repo (`node tests/*-smoke.js` умеют мокать, поэтому главное — ручной прогон живых сценариев: загрузка молдов, сохранение заказа, фото-аплоад, публикация каталога).
4. **Фриз + дельта (окно ~30 мин, согласовать с командой):** объявить фриз правок → финальный дельта-дамп → переключить: calc (`js/supabase.js`) → деплой (авто-SHA-бастинг уже донесёт мгновенно) → сайт env на Vercel → CI-секреты → прокси → timebot. Смоуки CI зелёные = готово.
5. **Rewrite URL файлов в данных:** скрипт заменяет старый домен на `db.recycleobject.ru` в mold_data/блобах (бэкап дампа до замены). Сайт `next.config` — добавить домен картинок.
6. **Figma-плагин:** передать Элине новый URL+ключ (одно место в конфиге плагина).
7. **Страховка:** старый Supabase живёт read-only **2 недели** (откат = вернуть URL одной строкой), затем пауза проекта. Бэкапы: с первой ночи.
8. **Фаза 2 (отдельно, после стабилизации):** домен calc с Vercel на Яндекс-зеркало (calc2 уже идентичен, DNS-переключение); сайт пока остаётся на Vercel.

## Критерии готовности
- Все смоуки CI зелёные на новом URL; calc/calc2/сайт/floor работают из РФ без VPN.
- Фото/glb/шаблоны открываются по новым URL; публикация каталога проходит end-to-end (сайт ≤3 мин, Figma по «Обновить каталог»).
- Ночной бэкап лежит в Object Storage, алерт в TG при падении healthcheck.
- Откат отрепетирован: возврат URL на старый проект < 5 минут.

## Риски
- **Дисциплина бэкапов** — главный риск self-hosted; закрываем автоматикой в день 1, не «потом».
- VM-ресурсы: если стеку тесно на 4 ГБ — вертикальный ресайз VM (минуты даунтайма).
- Забытый потребитель со старым URL — чеклист потребителей выше + старый проект в read-only ловит хвосты 2 недели (в его логах видно, кто ещё ходит).
