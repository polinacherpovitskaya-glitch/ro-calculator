# Производственная витрина цеха — публичная read-only страница (design)

- Дата: 2026-07-08
- Автор запроса: Полина (product owner)
- Образец: RePanel `https://web-production-d2761c.up.railway.app/production-plan-view` — публичная страница «План производства», которую уже отдают производству.
- Источник данных для зеркала: внутренний производственный календарь `calc.recycleobject.ru/#gantt` (`js/gantt.js` + `buildProductionSchedule()` в `js/calculator.js`).

## Проблема

Производственный календарь сейчас живёт **внутри** авторизованного калькулятора (`#gantt`). Сотрудникам в цехе нужно то же самое, что уже удобно сделано в RePanel: отдельная **публичная страница без входа**, которую можно кинуть в чат, открыть с телефона или повесить на экран в цехе. Они ничего не меняют — только смотрят: сколько людей в цехе, текущая мощность, очередь к запуску, что ждёт впереди и сколько часов займёт, плюс страница на каждый заказ (что делать: цвет, количество, фото и т.д.).

## Цели

- Публичная read-only **витрина** (зеркало `#gantt`), оптимизированная под **экран в цехе и телефоны** (адаптив, крупный шрифт).
- Наверху — сводка: **«Сейчас в цехе»** (люди), **«Текущая мощность»** (ч/день), **«В очереди»** (заказов / часов), риск-индикатор (первый день перегруза).
- **Календарь неделя/месяц** — как примерно расписаны заказы по дням, с этапами (Литьё / Сборка / Упаковка).
- **Очередь к запуску** — список заказов с дедлайном, часами (факт / план / осталось) и мелким превью (фото + точка цвета + количество).
- Секция **«Ждут молд / Китай»** и **«Требуют проверки»** — заблокированные заказы, как во внутреннем календаре.
- **Страница отдельного заказа**: календарь этого заказа **наверху**, ниже — фото, цвет, количество, состав (изделия / фурнитура / упаковка), этапы и часы, габариты/размер, дата отгрузки, маркировка/артикул, NFC-данные, примечание для цеха.
- Цифры **совпадают с `#gantt` до часа** — используется тот же движок расписания.
- **Актуальность:** автопубликация каждые ~15 мин + кнопка «Обновить».

## Не-цели (первая версия)

- Никакого редактирования, перетаскивания, смены очереди — только просмотр.
- Никакого входа/авторизации.
- Никаких live-запросов к БД со страницы (только опубликованный снимок).
- Никакого stage-level перепланирования.
- Файловые вложения заказа (кроме product-фото) — следующий шаг.
- Не трогаем version-анкеры calc (`js/version.json`, `app.js`, `index.html`) — это отдельная страница, не часть SPA калькулятора.

## Граница безопасности (что попадает в публичный JSON)

Ссылка публичная: открывается без входа у любого, у кого есть адрес (как RePanel). Поэтому безопасность обеспечивается **не** RLS, а **курированием на стороне публикатора** — в публичный JSON кладём только безопасные поля.

**Показываем:** название заказа, клиент (имя), дедлайн, статус, этапы и часы (план/факт/осталось), позиция в очереди, мощность, число людей в цехе, блокеры; по заказу — фото, цвет (название+hex), количество, состав, габариты/размер, дата отгрузки, маркировка/артикул, NFC-данные, примечание для цеха.

**Никогда не показываем:** цены, себестоимость, маржа, любые деньги; ключи Supabase/токены; контакты клиента (телефон/почта) — только имя.

## Архитектура (3 части)

```
Supabase ──► Публикатор (считает + курирует) ──► Яндекс Object Storage: /floor/*.json (+ фото)
                                                          │
                                          статическая витрина (fetch) ──► рендер
```

### 1. Публикатор снимка — `scripts/production-floor-publish.mjs`

Node-скрипт по образцу `scripts/fintablo-sync.mjs` / `scripts/tochka-sync.mjs`. Запускается по расписанию (GitHub Actions cron ~15 мин) с доступом к Supabase (service key из CI-секретов) и к Яндекс Object Storage (те же креды, что у существующего «Yandex static sync»).

Шаги:
1. Читает из Supabase те же данные, что и `#gantt`: `orders`, `order_items`, `settings` (`production_plan_state_json`, `planning_workers_count`, `planning_hours_per_day`, `production_holidays`), `china_purchases`, `time_entries`, `employees`, а также `product_templates` (фото по `template_id → photo_url`) и `app_colors` (названия/hex цветов).
2. Прогоняет **тот же** `buildProductionSchedule()` из `js/calculator.js`, загружая файл в Node через `vm` — **точно так же, как это уже делает `tests/production-calendar-smoke.js`**. Это гарантирует, что числа витрины = числа `#gantt`, без форка логики.
3. Собирает курированный JSON (только безопасные поля из раздела выше):
   - `floor/plan.json`: `{ generated_at, in_shop_count, daily_capacity, summary{ queue_count, queue_hours, first_overload_date, late_count, tight_count }, queue:[{ order_id, name, client, deadline, deadline_state, hours{plan,fact,remaining}, start_date, thumb_url, colors:[{name,hex}], quantity }], calendar:{ mode, days:[{date, is_nonworking}], segments:[{order_id, stage, start, span}] }, blocked:[{order_id, name, client, reason}] }`
   - `floor/orders/<id>.json`: `{ order_id, name, client, status, deadline, deadline_state, ship_date, marking, dimensions, weight, note, nfc:{...}|null, photo_url, colors:[{name,hex}], quantity, items:[{kind, name, colors, quantity, thumb_url}], stages:[{stage, plan, fact, remaining}], calendar_segments:[{stage, start, span}] }`
4. Фото: если `product_templates.photo_url` уже публично доступны — кладём ссылку как есть; если приватные — зеркалим картинку в `floor/photos/<id>.jpg`. Отсутствие фото — норма («фото если есть»).
5. Пишет файлы в бакет `calc2.recycleobject.ru` под префиксом `floor/`.
6. Идемпотентно; при любой ошибке загрузки/расчёта — **не публикует частичный результат**, оставляет предыдущий хороший снимок, шлёт алерт (переиспользовать существующий telegram-alert, `tests/send-telegram-alert-smoke.mjs`).

### 2. Публичный снимок (Яндекс Object Storage)

- Лежит под `calc2.recycleobject.ru/floor/…` (путь на текущем хостинге — выбранный вариант, отдельный домен не заводим).
- Статика, отдаётся с CDN. Кэш сбивается параметром `?ts=` при fetch (как в RePanel).

### 3. Статическая витрина — папка `production-floor/` (по образцу `corporate-gift/`)

- `production-floor/index.html` + `style.css` + `app.js`. Дизайн-система в стиле RePanel (`.public-*`, зелёный бренд `#214f2a`, крупный шрифт, скруглённые карточки). **Адаптив**: крупно на настенном экране, столбиком на телефоне.
- `index.html` фетчит `floor/plan.json` и рендерит: шапку с «Обновлено …» + «Обновить», сводку-карточки, календарь (неделя; месяц — опционально), очередь (карточки с превью/цветом/кол-вом), блокеры.
- Страница заказа — маршрут `#/order/<id>` (SPA-хэш, чтобы работало на статике без сервера) или `order.html?id=`. Фетчит `floor/orders/<id>.json`, рендерит календарь заказа наверху + все поля.
- Никаких секретов и обращений к БД со страницы.

## Поток данных

Supabase → публикатор (расчёт + курирование) → Яндекс `floor/*.json` (+ фото) → fetch со статической страницы → рендер. Кнопка «Обновить» = повторный fetch последнего снимка (не пересчёт).

## Обработка ошибок

- **Публикатор:** при ошибке — abort publish, сохранить последний хороший снимок, алерт в Telegram.
- **Страница:** если fetch не удался — показать «Не удалось обновить, показываем последнее» + повтор; пустые состояния мягкие («фото если есть», пустая очередь → «Очередь пуста»).
- **Устаревание:** если `generated_at` старше ~30 минут — ненавязчивая плашка «данные могли устареть».

## Тестирование

- `tests/production-floor-publish-smoke.js`: гоняет публикатор на фикстурных данных и проверяет:
  - в `plan.json` / `orders/<id>.json` **нет** ключей цен/себестоимости/маржи (список запрещённых полей);
  - числа очереди/часов совпадают с выводом `buildProductionSchedule()`;
  - `orders/<id>.json` содержит фото/цвет/кол-во/состав/габариты/дата отгрузки/маркировку/NFC/примечание.
- Переиспользовать `tests/production-calendar-smoke.js` для парити движка.
- (Опционально) Playwright-смоук статической страницы против фикстурного JSON: секции рендерятся, адаптив на mobile/desktop, страница заказа открывается.

## Выкладка

- Новая папка `production-floor/` + скрипт `scripts/production-floor-publish.mjs` + CI-cron публикатора + включение `production-floor/` и `floor/` в «Yandex static sync».
- Version-анкеры calc не трогаем.
- Отдать URL производству.
