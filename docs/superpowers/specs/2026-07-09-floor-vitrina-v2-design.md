# Витрина производства 2.0 — «что идёт сейчас / что ждём и почему» + фото

**Дата:** 2026-07-09
**Статус:** утверждён Полиной (дизайн), ожидает review спеки
**Область:** публичная read-only витрина цеха `calc2.recycleobject.ru/floor/`

## Задача

Полина (со слов производства): витрина должна чётко **разделять то, что идёт в моменте, и то, что ждём — и почему**. Плюс дать людям на производстве максимум наглядной информации, чтобы «легко и быстро собирать»: крупные цвета с фото, состав фурнитуры, и **фото-примеры**, которые менеджеры прикладывают внутрь заказа. И показывать, **где сейчас форма (молд)**, в т.ч. если она летит из Китая.

## Что реально есть в данных (аудит live 2026-07-09)

См. память `reference_order_data_shapes.md`. Кратко:

- **Этап производства** — из статуса заказа (`production_casting/printing/hardware/packaging`, `in_production`, `delivery`).
- **«Ждёт молд»** — из `order_items.item_data.base_mold_in_stock` (product-позиции) + статус `sample`. Это **единственный** сигнал по молду на заказе — булев флаг, **без ссылки** на конкретную закупку.
- **Фото-примеры** — `order_items.item_data.color_solution_attachment`: JSON-строка `{name,type,size,data:<base64>}` или массив таких. JPEG/PNG, ~60–440 КБ, только у `item_type: product`.
- **Цвета** — `app_colors.color_data.{name,number,photo_url}` (свотч-картинка, без hex).
- **Фурнитура** — `item_type: hardware`, есть имя/кол-во/вес/источник, **фото нет нигде** (ни у позиции, ни на складе, ни в закупке).
- **Формы/закупки в пути** — `china_purchases`: `status` {in_transit, received, delivered}; `purchase_data.{purchase_name, items[], status_history[], delivery_type, estimated_days, shipment_id}`. Деньги (`total_cny`, доставка) — приватные, не публикуем.

**Важное ограничение:** заказ **нельзя автоматически привязать** к конкретной летящей форме (у заказов нет кода `MRK…`, у закупок есть). Поэтому «где форма для ЭТОГО заказа» не выводим — вместо этого честный статус «Ждёт молд» на заказе + **отдельная** общая доска «Формы в пути».

## Дизайн

### 1. Доска (главный экран) — 4 блока сверху вниз

1. 🟢 **Сейчас в работе** — заказы, реально идущие в производстве. Показать **этап** (Литьё / Печать / Фурнитура / Упаковка) и остаток часов.
2. 🔵 **Очередь к запуску** — готовые (молд есть), ждут слот. Клиент · дедлайн · часы.
3. 🟠 **Ждут молд** — стоим и **почему**. Клиент · дедлайн.
4. ✈️ **Формы в пути из Китая** — **внизу**, отдельная сводка. По каждой летящей/едущей закупке: название формы, стадия (`на складе КНР → консолидация → летит`), тип доставки (авиа). **Без денег.** Источник — `china_purchases` со статусом `in_transit` (только реально едущие; `received`/`delivered` уже приехали и в блоке не показываются).

Разбиение «в работе» vs «очередь» vs «ждут молд» — это и есть требуемое разделение «идёт в моменте / ждём и почему».

### 2. Карточка заказа (на доске)

- Клиент · дедлайн · часы (**без денег**, как сейчас).
- **Цвета — крупными плашками** с фото-свотчем (сейчас мелкие; см. память `feedback_text_density`).
- **Фурнитура** — названия + количество (текстом; фото у фурнитуры в данных нет).
- **Фото-пример** — миниатюра `color_solution_attachment`, если есть.
- Вся карточка кликабельна → страница заказа (как уже сделано в PR #138).

### 3. Страница заказа `/floor/#/order/<id>`

- Календарь сверху (как есть).
- Полный состав: изделия, цвета (крупно), фурнитура, упаковка.
- **Все фото-примеры крупно** (галерея).

### 4. Обработка фото (техническое)

Фото-примеры — base64 60–440 КБ внутри `item_data`. Публиковать их inline в JSON нельзя (снимок разбухнет, медленно на цеховом экране). Publisher (`scripts/production-floor-publish.mjs`):

1. Достаёт base64 из `color_solution_attachment` (строка **или** массив).
2. Пишет каждую картинку файлом в бакет: `floor/photos/<order_id>-<item>-<n>.<ext>` (ext из `type`).
3. В `plan.json`/`orders/<id>.json` кладёт только **URL** на файл (как у цветов), не сам base64.
4. Дедуп по хэшу содержимого, чтобы не перезаливать неизменные фото. Идемпотентно.

Текущий `stripHeavy()` уже вырезает base64 — его логику надо согласовать: фото извлекаем в файлы ДО стрипа, остальные тяжёлые поля по-прежнему режем.

### 5. Приватность (граница безопасности)

Публикуем только: состав, статусы, часы, дедлайн, клиент (имя), цвета, фурнитура (текст), фото-примеры. **Никогда:** цены, маржа, себестоимость, реквизиты, ключи, суммы закупок. Аудит ключей публикуемого JSON — часть регрессии (как в существующих smoke).

## Модули (границы)

- **Движок готовности/этапа** (`js/production-core.js`) — уже даёт `production_ready_state` и статус; блок «Сейчас в работе» = заказы в активных production-статусах; «Очередь» = ready & not started; «Ждут молд» = blocked. Возможна доработка группировки, но без изменения правил готовности.
- **Publisher** (`scripts/production-floor-publish.mjs`) — добавляет: (а) сборку блока «Формы в пути» из `china_purchases`; (б) извлечение фото в файлы + URL. Каждое — отдельная функция с тестом.
- **Фронт витрины** (`production-floor/app.js`, `style.css`) — рендер 4 блоков, крупные цвета, фото-миниатюры/галерея.

## Тестирование

- Юнит: сборка «Формы в пути» из фикстуры `china_purchases` (статусы, без денег в выходе).
- Юнит: извлечение фото — строка и массив, разные `type`, дедуп по хэшу, отсутствие фото.
- Регрессия: аудит ключей публикуемого JSON — нет денег/PII/ключей/base64.
- Существующие floor-smoke остаются зелёными.

## Явно вне области (YAGNI)

- Привязка заказа к конкретной закупке/молду (нет данных — отдельный проект по вводу связи в калькуляторе).
- Фото по каждой SKU фурнитуры (нет данных).
- Редактирование/любые записи с витрины (остаётся строго read-only).
