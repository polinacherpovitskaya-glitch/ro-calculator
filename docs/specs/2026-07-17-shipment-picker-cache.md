# Быстрый picker в приёмке склада

**Дата:** 2026-07-17  
**Статус:** approved for implementation

## Проблема

Форма новой приёмки открывается до того, как повторно загрузит позиции склада.
При недоступном или медленном удалённом источнике форма остаётся без selector
позиций несколько секунд, хотя `Warehouse.allItems` уже заполнен на открытой
странице. Это мешает работе и делает live smoke нестабильным.

## Цель

Первый selector в новой приёмке должен рендериться из уже загруженного
кэша склада сразу. Повторная загрузка остаётся обычным путём для мест, где
кэша нет.

## Контракт

- `getItemsForPicker({ preferCurrentCache: true })` строит picker из
  непустого `Warehouse.allItems`, не ждёт сеть и не запрашивает reservations.
- Для кеша доступный остаток считается как `qty − reserved_qty`; если
  `reserved_qty` отсутствует, он равен нулю.
- Обычный вызов `getItemsForPicker()` сохраняет текущую live-загрузку и
  снимок reservations.
- Только экран новой приёмки выбирает быстрый cached-путь. Он не меняет
  данные склада и не делает запись в базу.

## Критерии приёмки

- При уже загруженной позиции форма приёмки сразу содержит `<select>` и
  доступную позицию, даже если новый запрос склада или reservations недоступен.
- Обычный picker продолжает получать свежие позиции и reservations.
- Изменение не затрагивает остатки, приёмку и ручную запись.

## Вне scope

- Изменение схемы склада.
- Автоматическая запись локального кэша в облако.
