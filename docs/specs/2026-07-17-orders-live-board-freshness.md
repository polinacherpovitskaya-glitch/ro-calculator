# Актуальные данные доски заказов

**Дата:** 2026-07-17  
**Статус:** approved for implementation

## Проблема

После пересчёта активного заказа его шапка в Supabase уже содержит новую
маржу, выручку и часы. Однако прямой сайт `calc.recycleobject.ru` при наличии
локального кэша возвращал его до запроса к базе. Поэтому доска могла показать
старую маржу, тогда как карточка открывала уже сохранённый расчёт.

## Решение

На прямом сайте с рабочим Supabase-подключением список заказов всегда сначала
запрашивает шапки заказов из общей базы и обновляет браузерный кэш ответом.
Локальные данные остаются запасным режимом только при ошибке/таймауте сети.

Статическое зеркало `calc2.recycleobject.ru` сохраняет текущий порядок:
локальный кэш и same-origin bootstrap, поскольку оно не должно полагаться на
прямой доступ к базе.

## Критерии приёмки

- Если локальная шапка содержит старую маржу, а Supabase — новую, доска
  `calc.recycleobject.ru` показывает новую маржу.
- При недоступной базе прямой сайт всё ещё открывает доску из локального
  кэша, не зависая.
- Статическое зеркало продолжает загружать список через bootstrap/кэш.
- Изменение не меняет финансовую формулу, JSONB-миграцию или данные заказов:
  оно устраняет только выбор устаревшего источника для списка.

## Вне scope

- Пересчёт цен, норм или уже сохранённых шапок заказов.
- Миграция `order_items.item_data` из `TEXT` в `JSONB`.
- Изменение поведения карточки заказа.
