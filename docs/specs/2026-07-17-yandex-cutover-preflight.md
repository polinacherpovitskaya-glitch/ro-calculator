# Preflight миграции Supabase в Yandex Cloud

**Дата:** 2026-07-17  
**Статус:** approved for implementation  
**Связанное решение:** перед cutover сначала исключаем риск неверных финансовых расчётов и делаем миграцию воспроизводимой.

## Проблема

Текущий дизайн миграции описывал целевую архитектуру, но не содержал воспроизводимого preflight-пакета: проверки `order_items.item_data`, SQL-миграции с rollback, измеримых stop-критериев и сценария проверки Storage/Figma. Это не позволяет безопасно назначать 30-минутный фриз.

Отдельный риск для финансов: в текущем Supabase `order_items.item_data` — `TEXT`. Даже когда содержимое валидно, PostgREST не может безопасно извлечь поля оператором JSON. Список заказов поэтому до миграции должен показывать сохранённые показатели, а не неполный живой пересчёт.

## Цели

- Сделать read-only аудит содержимого `item_data` повторяемым и безопасным.
- Подготовить транзакционный SQL-переход `TEXT -> JSONB` с полным backup и отдельным rollback.
- Зафиксировать runbook репетиции: данные, Storage, calc/calc2, сохранение заказа, фото и Figma-плагин.
- Определить явные go/no-go критерии до назначения фриза.
- Не менять боевую БД, URL, ключи, Storage или Figma в рамках этого пакета.

## Требования

1. Аудит обязан читать только `id`, `order_id`, `item_data`; не выводить сами данные и не менять записи.
2. Аудит считает безопасными только JSON-объекты (нативный JSONB или JSON-объект в TEXT). Пустые, битые и double-encoded значения блокируют миграцию.
3. SQL обязан работать одной транзакцией, создавать неизменяемую копию исходного `item_data`, проверять тип и данные до `ALTER`, а после миграции — тип и JSON-форму.
4. Rollback обязан возвращать исходный `TEXT` из backup, а не сериализовать новое JSONB-содержимое как «похожий» текст.
5. Runbook обязан отличать подготовку/репетицию от самого cutover и не содержать ложных команд для ещё не созданной инфраструктуры.
6. Перед cutover должны быть зелёными: parity-фикстуры, сохранение+reload тестового заказа на новом стенде, upload+reload фото на calc2-копии, inventory Storage, URL-rewrite dry-run, Figma-каталог после смены конфигурации.

## Вне scope

- Выполнение `ALTER` в production.
- Переключение `SUPABASE_URL`, anon key, API Gateway, Vercel или DNS.
- Перенос файлов в новый Storage и массовое изменение ссылок.
- Изменение Figma-файла или передача ключей третьим лицам.
- Любая очистка старого Supabase.

## Критерии приёмки

- Read-only аудит на production подтверждает, что все `item_data` готовы к переходу, либо возвращает ненулевой exit code и безопасный отчёт.
- SQL и rollback содержат backup, проверки и транзакционные границы; их контракт покрыт smoke-тестом.
- Есть актуальный runbook с ролью фриза, preflight, rehearsal, cutover, validation и rollback.
- Обновлены общий план, статус и test-plan, чтобы следующий запуск мог продолжить работу без истории чата.
