# Согласованное сохранение пересчитанного заказа

**Дата:** 2026-07-17  
**Статус:** approved for implementation

## Проблема

Автосохранение и явное «Сохранить» формируют отдельные снимки одного заказа и выполняют сетевые действия асинхронно. Отмена таймера не останавливает уже начатое автосохранение: после подготовки файлов оно может записать устаревшие расчётные показатели и состав заказа позже ручного сохранения.

На calc2 есть второй путь возврата старых цифр: успешное сохранение обновляет localStorage без свежего `updated_at`, а фоновый статический bootstrap приходит с задержкой. Из-за этого bootstrap может ошибочно признать старый заказ новее и заменить локальные шапку и позиции. После обновления страницы пользователь снова видит прежний снимок.

Это особенно заметно после изменения затратного параметра, например дополнительного молда: в калькуляторе вручную сохранена новая маржа, но список заказов после reload возвращается к прежней.

## Цели

- Ручное сохранение всегда является последней записью пользователя для текущего состояния формы.
- Автосохранение, которое стало устаревшим после начала ручного сохранения, не записывает ни шапку заказа, ни позиции.
- Если запись автосохранения уже ушла в сеть, ручное сохранение ожидает её и записывается после неё.
- Не очищать локальный снимок и не показывать «Черновик сохранён» от устаревшего автосейва.
- Не позволять устаревшему calc2 bootstrap заменять только что сохранённые шапку или позиции заказа.

## Требования

1. Все вызовы `saveOrder(order, items)` из `Calculator` для одного экземпляра формы проходят через одну последовательную очередь.
2. В начале ручного save инкрементируется поколение сохранения и отменяется ожидающий таймер автосейва.
3. Автосейв захватывает поколение перед асинхронной подготовкой. Перед постановкой записи в очередь и перед обновлением UI он проверяет, что поколение всё ещё актуально.
4. Reset формы также инвалидирует уже начатое автосохранение.
5. Очередь продолжает принимать следующие записи после сетевой ошибки предыдущей.
6. Regression-smoke воспроизводит оба порядка событий:
   - уже начатый autosave, затем manual save: manual write выполняется последней;
   - autosave ждёт асинхронную подготовку, manual save завершён: старый autosave не пишет базу.
7. Успешный `saveOrder` записывает тот же write timestamp в локальную копию шапки. При bootstrap более свежая локальная шапка удерживает весь свой набор позиций, включая удалённые/изменённые строки.

## Нефункциональные ограничения

- Не менять публичные API `Calculator.recalculate()`, `Calculator.toJSON()` и `Calculator.loadOrder()`.
- Не менять схему `orders`/`order_items` и не выполнять миграций БД.
- Не переписывать `js/app.js` или `js/supabase.js` целиком: только локальная координация записи в Calculator.
- Сохранить существующий статус workflow и механизм аудита изменений.

## Критерии приёмки

- После ручного «Сохранить» обновление списка/страницы показывает именно сохранённые показатели и состав.
- Старый автосейв не может перезаписать ручной save, даже если его promise завершается позже.
- Старый bootstrap calc2 не может вернуть прежнюю маржу или состав сразу после ручного save.
- `node tests/order-flow-smoke.js` и `node tests/version-smoke.js` проходят.
- Версия приложения и cache-bust `app.js` подняты синхронно.

## Вне scope

- Пересчёт исторических заказов при изменении текущих норм.
- Миграция `item_data` из TEXT в JSONB и cutover новой БД.
- Изменение формулы себестоимости или маржи.
